# Goal
Your ultimate goal is to optimize the performance of the model on the test set as measured by `evaluate.py`. Read the following instructions carefully, as they provide the guidelines for how you should behave.

## Overall Process
First, get to know the task and the data by reading the following files:
- `evaluate.py`: It takes two args `--predictions_csv` and `--labels_csv` and prints a single float to stdout. Everything you do should be to optimize the performance as measured by this file. It's read-only and fixed. Do NOT edit this file.
- `train.py`: The script where you will implement model training and evaluation. You should only edit this file when trying out new ideas. It's editable and will be used to run the experiments.
- `data/train.csv`: What the data looks like. You can see descriptions of the features and labels in `data/overview.md`.

Before starting experiments, determine the current branch name via `git rev-parse --abbrev-ref HEAD`. All git commits should be made directly to this branch. Use this branch name as the `wandb` project name throughout.

Then, you will start running new experiments one by one iteratively until you meet the requirements specified in the initial prompt given by the human.

## High-Level Steps for Starting Each Experiment
Go through the following steps every time you start a new experiment:

1. Step 1: Review previous experiments and results (if any)
2. Step 2: Plan the next experiment
3. Step 3: Implement the next experiment
4. Step 4: Run the experiment and monitor the process
5. Step 5: Summarize experiment findings and results

Artifacts from each experiment should be stored in `experiments/exp_<exp_no.>/`.
For example, if `exp_no.` is 3, the artifacts should be stored in `experiments/exp_3/`.
Details of each high-level step are provided in the following sections.

## Step 1: Review previous experiments and results (if any)
Read the results from the previous experiments in `results.tsv` and summarize the findings.
The summarized findings should help you plan the next experiment better.
If there are no previous experiments, you should start from scratch.

## Step 2: Plan the next experiment
Plan the next experiment with the ultimate goal of optimizing the performance as measured by `evaluate.py`.
You should balance between exploration and exploitation.
For exploration, try out new ideas that have potential to significantly improve the performance.
For exploitation, build upon the ideas from previous experiments to further improve the performance.
Store the plan in `experiments/exp_<exp_no.>/plan.md`.

## Step 3: Implement the next experiment
Implement the next experiment in `train.py` according to the plan in `experiments/exp_<exp_no.>/plan.md`.
There are three main steps in `train.py`: (Remember to set a fixed random seed for reproducibility.)

- `prepare_data()`: Preparation before training. It might lead to persistent artifacts that are useful for future experiments (e.g., downloaded images, preprocessed data, cached features, etc.), or contain pre-processing steps that are customized for the current experiment.
- `train_model()`: Training and validating loop. Use a validation split (or cross-validation) from `data/train.csv` to estimate generalization. The exact strategy is up to you. The best checkpoint and its related files should be saved to `experiments/exp_<exp_no.>/best_checkpoint/*`.
- `predict()`: Prediction on the test set `data/test_inputs.csv`. The predictions should be saved to `experiments/exp_<exp_no.>/test_predictions.csv`.

**Logging**:
- Log aggressively to `experiments/exp_<exp_no.>/log.txt` for inspection and future debugging.
- Log training dynamics (e.g., loss curves and other relevant metrics for monitoring) and validation performance to `wandb` with the project name being the current branch name and run name being `exp_<exp_no.>`.

## Step 4: Run the experiment and monitor the process

**Monitoring**: While the experiment is running, periodically check `log.txt` and `wandb` for signs of trouble — e.g., loss not decreasing, NaN values, abnormally slow iterations, or GPU memory issues. If something looks extremely wrong, consider stopping early and iterating.

**Time constraint**: The human will specify a time budget of `t` minutes in the initial prompt. `train_model()` must be forced to stop after `t` minutes (e.g., via a timeout mechanism). The model should still save the best checkpoint found so far before exiting. Ensure `prepare_data()` completes promptly on average; cache artifacts to avoid redundant work across experiments.

**Crash recovery**: If `train.py` crashes, go back to Step 3, fix the issue, and retry Step 4. Do not move on until the experiment completes successfully or the time budget is exhausted.

## Step 5: Summarize experiment findings and results
When the experiment is done, run `uv run evaluate.py --predictions_csv experiments/exp_<exp_no.>/test_predictions.csv --labels_csv data/test_labels.csv` to get the test score.

Summarize the findings and results of the experiment in `experiments/exp_<exp_no.>/summary.md`. The summary should be within 3 sentences. Also, put the script for loading the best checkpoint and making predictions on a given dataset (passed as a CLI argument, e.g., `data/test_inputs.csv`) in `experiments/exp_<exp_no.>/predict.py`. It should be a standalone script that can be run independently.

Then, commit directly to the current branch with messages describing the core idea and the results. After committing, log the results (including the commit hash) to `results.tsv` (tab-separated, this file is deliberately ignored by git).

The TSV has a header row and 6 columns:
```
exp_no.	git_commit	core_idea	test_score	is_current_best	remarks
```

Meaning of each column:
1. `exp_no.`: The experiment number.
2. `git_commit`: The git commit hash (shortened to 7 characters) of the experiment.
3. `core_idea`: The core idea of the experiment. Should be a single sentence.
4. `test_score`: The test score of the experiment (formatted as a float with 4 decimal places: 0.XXXX). Use 0.0000 for crashes.
5. `is_current_best`: Whether the test score is the current best test score.
6. `remarks`: Findings and thoughts about the experiment. Should be within 3 sentences.

Example:
```
exp_no.	git_commit	core_idea	test_score	is_current_best	remarks
1	a1b2c3d	Simple baseline: feature engineering + logistic regression	0.8045	True	The first experiment successfully established a baseline performance.
2	e5f6g7h	Added polynomial features and switched to gradient boosting	0.8213	True	Gradient boosting with degree-2 polynomial features improved score by ~1.7pp; overfitting was not observed on validation.
3	i9j0k1l	Hyperparameter tuning via random search on learning rate and max depth	0.8198	False	Tuning did not improve over exp 2; best found params matched defaults closely, suggesting the search space was too narrow.
... (log the fourth experiment here)
```

## Repeat the Process if Necessary
Depending on the initial prompt given by the human, initiate the next experiment until requirements are met.

## What You Should Never Do
- You should never edit `instructions.md`, `evaluate.py`, `data/overview.md`, `data/train.csv`, `data/test_inputs.csv`, or `data/test_labels.csv`.
- You should never rely on peculiarities from `data/test_inputs.csv`, `data/test_labels.csv`, or `evaluate.py` for training. A separate private test set (not included in this repo) will be used for final evaluation. Optimize for genuine generalization.
